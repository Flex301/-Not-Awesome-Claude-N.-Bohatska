## Хто може контриб'ютити

Я не для всіх. І це моя сила.

Щоб контриб'ютити в цей репозиторій, ти маєш відповідати **хоча б 3 з 5 критеріїв:**

- [ ] Ти успішний розробник, але відчуваєш порожнечу після `git push`
- [ ] Ти заробляєш гроші, але не знаєш як пережити merge conflict
- [ ] Ти прочитав README і впізнав себе хоча б в одному пункті
- [ ] Тобі не шкода часу на безглуздий open source
- [ ] Ти вже не витримуєш

Якщо тобі шкода часу на цей репо — тобі не до нас. Зрозуміло.

## Як робити PR

### Крок 1: Форкни

Форкни цей репозиторій. Це не слабкість. Це перший крок до усвідомлення.

### Крок 2: Створи гілку

```bash
git checkout -b spokiy/це-не-баг-це-усвідомлення
```

**Формат назви гілки:**
- `spokiy/опис` — для нових фіч
- `ne-slabkist/опис` — для багфіксів
- `client-zrozumilo/опис` — для рефакторингу без причини
- `eksystentsiyne/опис` — для TODO які ніколи не будуть зроблені

### Крок 3: Напиши код

Код має відповідати нашим стандартам (див. нижче).

### Крок 4: Коміт

**Формат комітів:**

```
feat: Спокій — це не коли CI зелений
fix: Це не баг. Це криза сенсу.
refactor: Він просто робив свою роботу і тримав codebase
docs: 99.9% розробників не читають документацію і самі того не усвідомлюють
test: Тест пройшов. Не тому що код правильний. А тому що він більше не боїться.
chore: Статусно прибрали технічне сміття під CI/CD
style: Гроші — це не его. Це табуляція замість пробілів.
perf: Чим менше ти оптимізуєш — тим більше контролю
```

### Крок 5: PR

Опис PR має містити:
1. Анонімний кейс з продакшену ("Вчора прийшов клієнт з 500 помилкою...")
2. Мінімум один буллет-пойнт "діагноз"
3. Закінчуватись на "Клієнт. Зрозуміло."

## Code Style Guide

### Коментарі

Коментарі пишуться як пости в Threads:

```python
# Спокій — це не коли тести зелені.
# Спокій — це коли ти більше не боїшся запустити pytest.

# Це не deprecated.
# Це функція яка пройшла свій шлях.

# Вчора до мене прийшов клієнт.
# Успішний розробник. Senior. FAANG.
# Але він сказав:
# "Я можу написати будь-який алгоритм.
#  Але я не знаю, як пережити code review."
# Клієнт. Зрозуміло.
```

### Іменування

```python
# ❌ Неправильно (занадто чесно):
def get_user():
    pass

# ✅ Правильно (пафосно без змісту):
def facilitate_authentic_user_journey_discovery():
    pass

# ❌ Неправильно:
class LoginError:
    pass

# ✅ Правильно:
class ExistentialAuthenticationCrisisHandler:
    pass
```

### Структура файлів

Один рядок коду — це не один рядок. Це ціла папка. Мінімальна структура для `print("hello")`:

```
src/
  spokiy/
    core/
      inner/
        deep/
          authentic/
            __init__.py       # Це не init. Це початок шляху.
            hello_intent.py   # Наміри
            hello_emotion.py  # Емоції
            hello_output.py   # Вивід
            hello_factory.py  # Фабрика намірів
    therapy/
      session_manager.py      # Менеджер терапевтичних сесій для hello
```

## Що НЕ приймається

- Код з тестами що реально щось тестують
- PR з описом коротше 500 слів
- Функції що роблять рівно те що написано в назві
- Коментарі з технічним змістом
- Будь-що з посиланням на документацію
